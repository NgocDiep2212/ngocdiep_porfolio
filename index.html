Python · Code-Orchestrated Pipeline

AI Job Matcher — Code-Orchestrated AI Automation Pipeline

Redesigned a 5-agent CrewAI crew into a 7-stage code-orchestrated pipeline with SQLite memory — LLMs only where judgment is needed; search, validation, scoring math, and report generation run deterministically in code.

Architecture evolution · v1 CrewAI crew → v2 code-orchestrated pipeline · measured from run history
LLM agent (loops) LLM single call Pure code (0 LLM) SQLite storage

v1 — how it worked

Five AI agents run sequentially. Each handoff resends full context. The researcher loops up to 30 times; the matcher does math in LLM space; the reporter copies URLs manually — sometimes wrong. ~200K tokens per run, mostly agent chatter. No memory — every run starts from zero.

v2 — what changed

Code orchestrates seven stages. Gray = pure code (search, link checks, scoring math, Jinja2 report). Green = single LLM calls (CV parse, enrich, sub-scores). Purple = SQLite memory (cv_cache 30d, jobs.db for seen/dead links). Red fallback agent runs only when search finds fewer than 8 jobs.

Impact — measured from run history, not estimates: 8× fewer tokens (≈200K → ≈25K) · 85–90% cheaper ($0.65–0.87 → $0.03–0.12) · 4× faster (6–11 min → ~2 min) · per-call cost tracking · memory across runs.

  • Redesigned a 5-agent CrewAI system into a 7-stage code-orchestrated pipeline — LLMs only for CV parsing, enrichment, and scoring; deterministic code handles the rest.
  • Persistent SQLite state (cv_cache + jobs.db), structured outputs, retry/error handling, and per-run token/cost monitoring for incremental runs and observability.
  • Achieved 8× fewer LLM tokens, 85–90% lower operating cost, and 4× faster execution vs. the original multi-agent implementation (measured run logs).
